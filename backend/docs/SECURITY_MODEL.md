## Authentication

### JWT Tokens

- **Algorithm**: HS256 (`backend/config.py:126`)
- **Secret key**: `JWT_SECRET` env var. **Default is a well-known placeholder ‚Äî MUST be overridden in production.**
- **Access token expiry**: 900 seconds (15 minutes) (`config.py:127`)
- **Refresh token expiry**: 2,592,000 seconds (30 days) (`config.py:128`)
- **Access token claims**: `user_id`, `email`, `role`, `exp`, `iat`, `session_id`, `type: "access"`
- **Refresh token claims**: same but with `"type": "refresh"`
- **Validation**: `get_auth_context` rejects tokens with `type: "refresh"` to prevent refresh-as-access abuse

### Session Management

Sessions stored in `user_sessions` table. Each session identified by `session_id` embedded in JWT.
- Password change (`POST /auth/change-password`) invalidates all active sessions.
- Source: `backend/domain/services/auth_service.py`

### API Keys

API keys stored as hashes in `api_keys` table. Users manage via `/auth/api-keys` endpoints (create, list, delete).

### Password Hashing

- **Algorithm**: PBKDF2-HMAC-SHA256 (`backend/database/auth/utils.py:15-75`)
- **Iterations**: 100,000
- **Salt**: 32 bytes via `secrets.token_bytes(32)`
- **Comparison**: `hmac.compare_digest()` (constant-time)
- **Architecture**: `PasswordHasherPort` ‚Üí `PBKDF2PasswordHasher` adapter (`backend/adapters/password_hasher.py`)
- **Note**: PBKDF2 is acceptable per OWASP ASVS V2.4.4, but bcrypt or argon2id would be preferable.

### Account Lockout

After 5 consecutive failed login attempts, the account is locked for 30 minutes.
Source: `backend/domain/services/auth_service.py`

## Authorization (RBAC)

### Implementation

`backend/utils/security/rbac.py` provides `RBACDependency`, a singleton `rbac` instance used as a FastAPI dependency.

Available dependency methods:
- `rbac.require(permission)` ‚Äî requires a specific permission string
- `rbac.require_any(*permissions)` ‚Äî requires at least one of the listed permissions
- `rbac.authenticated()` ‚Äî requires a valid session, any authenticated user
- `rbac.authenticated_read_only()` ‚Äî authenticated with no write permissions
- `rbac.verified()` ‚Äî authenticated + email verified
- `rbac.optional()` ‚Äî auth is optional; returns `AuthContext` or `None`
- `rbac.public()` ‚Äî no auth required

### RBAC Port

Admin RBAC operations go through `RBACPort` ‚Üí `RBACAdapter` (`backend/ports/rbac.py`, `backend/adapters/rbac_adapter.py`). Wired in `api_main.py`.

### Permissions

Permission strings follow `<resource>:<action>` pattern:
- `admin:read` ‚Äî read access to admin endpoints
- `admin:write` ‚Äî write access to admin endpoints
- `products:read` ‚Äî read access to product data

Permissions assigned via `RolePermission` (per-role) and `UserPermission` (per-user) tables.

**Critical**: The `ADMIN` role does NOT automatically have all permissions ‚Äî must be explicitly assigned.

### Access Control

`AuthContext.can_access_product()` at `backend/domain/entities/auth_models.py:287-301` now returns `False` (deny-by-default) for non-admin users without `products:read:all`. Full ownership check (querying `product_tracks` by `user_id`) is not yet implemented but the default is secure.

## Network Security

### CORS Policy

Source: `backend/config.py:173-174`

- **Default origins**: `["*"]` (wildcard) ‚Äî **MUST be restricted in production**
- **Credentials**: `allow_credentials=true` (default)
- **Override**: Set `PHOVEU_BACKEND_CORS_ORIGINS` to comma-separated origin list

### Rate Limiting

Source: `backend/utils/middleware/rate_limiting.py`

- **Default**: 20 requests per 60-second window
- **Storage**: Redis sorted sets when available, in-memory dict fallback (per-process)
- **Bypass**: Configurable paths via `RATE_LIMIT_BYPASS_PATHS` and key via `RATE_LIMIT_BYPASS_KEY`
- **Per-endpoint overrides**: `/bench`, `/auth/token`, `/auth/register` have configurable limits (`config.py:214-224`)
- **Note**: With Redis, rate limits are shared across all workers. Without Redis, falls back to per-process in-memory.

### Product-Add Rate Limiting (Tiered)

Source: `backend/utils/middleware/product_add_rate_limiter.py`

Separate per-user rate limiter for `POST /products/add`:
- **Unverified users**: `TRACK_RATE_LIMIT_UNVERIFIED` requests per window (default 10)
- **Verified users**: `TRACK_RATE_LIMIT_VERIFIED` requests per window (default 50)
- **Window**: `TRACK_RATE_LIMIT_WINDOW` seconds (default 60)
- **Storage**: Redis sorted sets with in-memory fallback (same pattern as global rate limiter)
- **Response headers**: `X-Track-RateLimit-Limit`, `X-Track-RateLimit-Remaining`, `X-Track-RateLimit-RetryAfter`
- **On exhaustion**: Returns HTTP 429 with `retry_after` in JSON body

## SMTP / Email Security

Source: `backend/adapters/hooks/email_handler.py`

Three SMTP modes: SSL, STARTTLS, and plain (unauthenticated).
Plain SMTP transmits credentials in cleartext ‚Äî only use on trusted internal networks.

DEBUG log statements in the email handler have been converted to `logger.debug()` with lazy `%s` formatting ‚Äî they no longer appear in production logs unless debug logging is explicitly enabled.
